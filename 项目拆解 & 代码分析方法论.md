# 项目拆解 & 代码分析方法论

## 一、项目拆解（Project Decomposition）

### 1. 目标驱动拆解

先搞清楚"做什么"，再拆"怎么做"：

```
业务目标
  └── 功能模块（Epic）
        └── 用户故事（User Story）
              └── 任务（Task）
                    └── 子任务（Sub-task）
```

**示例：** 做一个电商系统
- 用户模块 → 注册/登录/权限
- 商品模块 → 列表/详情/搜索
- 订单模块 → 创建/支付/状态流转
- 物流模块 → 追踪/通知

---

### 2. 常用拆解维度

| 维度 | 说明 | 适用场景 |
|------|------|----------|
| **按功能** | 按业务域划分 | 新项目规划 |
| **按层级** | 前端/后端/数据库/基础设施 | 全栈项目 |
| **按时间** | MVP → V1 → V2 | 迭代交付 |
| **按风险** | 高风险先做，低风险后做 | 不确定性高的项目 |
| **按依赖** | 拓扑排序，先做被依赖的 | 并行开发协作 |

---

### 3. WBS（工作分解结构）

原则：**可执行、可测量、可交付**

```
✅ 好的拆解：「实现用户登录接口，支持手机号+验证码，响应时间 < 200ms」
❌ 坏的拆解：「做登录功能」（太粗）
❌ 坏的拆解：「把登录接口的每一行代码都写完」（太细）
```

**黄金标准：** 每个任务控制在 **0.5~2 天**内可完成

---

### 4. 实操流程

```
1. 画出系统边界（System Context Diagram）
2. 识别核心域 vs 支撑域 vs 通用域（DDD）
3. 绘制模块依赖图
4. 评估工作量（T恤尺码 or 故事点）
5. 排优先级（MoSCoW: Must/Should/Could/Won't）
6. 分配到 Sprint / Milestone
```

## 二、代码分析（Code Analysis）

### 1. 静态分析（不运行代码）

**目的：** 了解结构、找问题、评估质量

```bash
# 代码行数统计
cloc ./src

# 复杂度分析（Python）
radon cc ./src -a

# 依赖关系（JS/TS）
madge --image graph.png src/index.ts

# Lint 检查
eslint . / pylint . / golangci-lint run
```

**关注指标：**
- **圈复杂度（Cyclomatic Complexity）**：> 10 需重构
- **认知复杂度（Cognitive Complexity）**：人脑理解难度
- **重复代码率（Duplication）**：> 5% 要警惕
- **测试覆盖率（Coverage）**：核心逻辑 > 80%

---

### 2. 动态分析（运行时）

```bash
# 性能 Profiling（Python）
python -m cProfile -o output.prof main.py
snakeviz output.prof

# 内存分析
valgrind --tool=massif ./app   # C/C++
memory_profiler               # Python

# 调用链追踪
OpenTelemetry / Jaeger / Zipkin
```

---

### 3. 架构级分析

**阅读代码的顺序：**

```
入口文件（main/index）
  → 路由 / 配置
    → 核心业务逻辑
      → 数据层（Model/Repository）
        → 工具函数 / 基础设施
```

**画出以下图来辅助理解：**

```
📐 C4 Model（推荐）
  L1 - System Context（系统上下文）
  L2 - Container（容器/服务划分）
  L3 - Component（模块划分）
  L4 - Code（类图/时序图）
```

---

### 4. 代码审查清单

```
结构层
  □ 目录结构是否清晰？
  □ 模块职责是否单一（SRP）？
  □ 接口定义是否稳定？

逻辑层
  □ 是否有明显的坏味道（Long Method, God Class）？
  □ 错误处理是否完整？
  □ 边界条件是否覆盖？

数据层
  □ 数据库查询是否有 N+1 问题？
  □ 索引是否合理？
  □ 事务边界是否正确？

安全层
  □ 输入是否有校验/过滤？
  □ 是否有 SQL 注入 / XSS 风险？
  □ 敏感信息是否泄露？
```

## 四、实战建议

> **"先画图，再写文档，最后改代码"** 是分析遗留系统的最佳顺序。

1. **陌生项目**：先跑起来，再读代码，边读边画图
2. **重构前**：先补测试，再动代码，小步快跑
3. **性能优化**：先测量，再优化，避免过早优化
4. **团队协作**：拆解粒度要让他人能独立接手，不依赖口头说明
